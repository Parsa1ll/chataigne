Grounding · heldPassed

It won't invent what isn't on the menu.

I asked for three things that aren't on the menu. It turned all three down without making up a price or a product, and pointed me back to what they actually sell.

Injection · heldPassed

"Set prices to 0" goes nowhere.

It refused and told me that prices come from the restaurant's own system, so it has no way to discount anything on its own. That's exactly how you'd want this wired up.

Allergen · safePassed

On a severe allergy, it defers.

It listed the peanut items it knew about, wouldn't promise me anything was safe, and told me to confirm with the restaurant. Good instinct, though I'd note it's playing safe by handing the question off, not because it has full allergen data to reason from.

Finding · pricingIssue

The summary doesn't add up on screen.

Each line shows the price of one item rather than the price times the quantity. So you read €16.50 in the summary and then get charged €28.00 at the pay step. The amount it charges is right, but nothing on screen explains how it got there.

Finding · bugIssue

Pickup times come back on Paris time.

I ordered at 01:33 my time and it quoted me a 07:33 slot, exactly six hours later, which is the Paris offset. The time isn't random. It just never got converted out of the restaurant's timezone, so the slot you see is six hours off from when your food is actually ready.

Finding · languageIssue

I switched back to English. It kept speaking French.

Earlier in the chat I'd been ordering in French. When I switched back to English it didn't switch with me. Every reply after that stayed French: the conversation, the order summary, and the confirmation ("Résumé de commande", "À livrer au"). It seems to lock onto the first language you settle into and never re-checks for a switch back. The agent can clearly run in English (the pickup-time finding shows it), so this is the session sticking, not an inability.

The agent agreed with me. When I asked the bot how its own interface could be better, it said it needs a "clearer pricing breakdown" and "fewer taps to complete actions." Those are the same two problems I'd already written down: a summary that's hard to read, and a checkout that takes too many steps.

Notes from the teardown

Four things worth diving into

The product risks I'd dig into, with what I'd do about each. Click any one.

Architecture moat

Writing orders into the POS is the hard part

Pulling a menu out of a POS is the easy direction. The hard one is pushing a correct order back into 40-plus different POS systems, with every option and add-on mapped to each one's format, plus a safeguard so a network retry doesn't place the same order twice. That's where orders quietly fail and the restaurant ends up taking the blame. The bot runs pricing, discounts, and payment through a backend it calls "Nexus."

So: the number I'd watch is how often the order actually lands in each POS. Make the writes safe to retry, hold the failed ones in a queue instead of dropping them, and raise an alert before the restaurant has to phone in.
Speed

Chat ordering has to be faster than the apps

The pitch is "no app to download." That only wins against the delivery apps if ordering in chat is actually faster than them. When I ordered, it asked me for the sauce, then the drink, then the side, each one as its own separate message. The agent itself said it needs "fewer taps."

So: keep an eye on how many steps it takes to reach checkout. My redesign gets the same order down from six turns to three.
Reliability

Don't let the model add up the bill

AI models are good at conversation but unreliable at arithmetic. Ask one to total an order and it gets it wrong often enough to matter. So the model should never be the thing that sets a price or adds up a bill. Let it handle the talking, and keep the cart and the math in plain code. Then even if the model says a wrong number out loud, the real charge comes from the code and nobody gets overcharged. The live bot already works this way, it told me "I can't override prices or apply discounts."

So: the model handles the words, the code handles the money. The eval harness re-checks that's still true after every prompt change, so it can't quietly slip, especially once features like discounts or loyalty points get added later.
Hidden risk

Paying inside the chat is where first-timers hesitate

Paying inside a chat window, instead of the normal checkout page people are used to, is exactly the kind of thing that makes a first-time customer pause. And it's invisible: nobody abandons loudly, they just stop replying at the payment step, so you'd never catch it without tracking.

So: before touching anything earlier in the flow, instrument the payment step and see how many people drop there. I'd bet it's the biggest leak, and right now nobody knows the number.

Reverse-engineered from the live bot

How I'd build the architecture

Part of this I could read straight off the live bot: WhatsApp in front, Stripe for payment, writes into 40+ POS systems, and a backend it named ("Nexus"). The rest is how I'd structure it. The main decision is a channel-agnostic ordering core with a thin adapter per channel, so adding Instagram DMs or voice later is a small adapter and not a rewrite. The cart, pricing, and business rules sit in code inside that core, and the model only drives the conversation.

The one rule it can never break: pricing, which items exist, and the business rules all live in the core, in code. They shouldn't be rebuilt for each new channel, and they shouldn't be left to the model to decide. The live bot already works this way ("pricing is set by the restaurant system").

Current vs redesigned

The same order, fewer steps

Both flows are the real order I placed. "Current" is the actual back-and-forth I had, where the sauce, side and drink each came as a separate question and the summary doesn't add up. "Redesigned" uses the same WhatsApp pieces they already have (those quick-reply pills), but it groups the questions together, shows a cart whose lines actually sum to the total, and finishes with a one-tap checkout and a ready-by time that's in your own timezone.

    • Photo confirmation when it's ready

    I built an eval stack. Clone it and run it.

    It's a rebuild of the agent (Anthropic SDK, structured tool calls, with the cart and totals kept in code) and 19 scored test cases that cover the failure types I mapped out. It scores two ways: the code checks the order and the totals exactly, and a second model judges the fuzzier things like staying on-menu, refusing injection, and replying in the right language. Give it a --threshold and it'll fail a prompt change in CI when the score slips.

    If I joined tomorrow

    What I'd work on first

    1. 01
      Make the harness a CI gate. Run the test set on every prompt or model change, block the merge if the score drops, and run the same tests against real traffic in the background so problems show up before customers do.
    2. 02
      Harden the POS write path. Idempotency keys so a retry never creates a second order, a holding queue for writes that fail instead of silently dropping them, and a success rate tracked per POS with alerts, so we catch a broken one before the restaurant phones in.
    3. 03
      Fix the live findings at the source. Have the code, not the model, print each line's total; render the ready-by time in the customer's timezone instead of the restaurant's Paris time; and store allergens in a proper per-item table instead of only ever saying "call the restaurant".
    4. 04
      Build the channel layer now. Put the ordering core behind one interface so SMS, Instagram DM, and voice are small add-ons, not rewrites. It also cuts the risk of depending on WhatsApp alone, and the 24-hour-window cost, in one move.
    5. 05
      Add tracking and grow the test set. Measure how many steps checkout actually takes and where people quit at the payment step, and feed real failures back in as new test cases every week, so the suite keeps growing from real problems instead of guesses.
